# 阶段⑤ — 给数据治理 wiki 挂 RAG

学习路径第 5 阶段。真正解决最初目标"让模型更懂数据治理"：
**准确回答、给真实出处、wiki 更新即生效**。这正是 RAG 的主场（微调不是）。

## RAG 是什么

RAG = 检索增强生成（Retrieval-Augmented Generation）。提问时：
1. 先从 wiki 里**检索**出最相关的几段原文
2. 把它们连同问题一起喂给大模型
3. 模型**基于这些真实原文**作答

所以答案有据可依、能附出处、改 wiki 不用重训——和"靠模型记忆"的微调截然不同。

## 怎么跑

```bash
python stage5/build_index.py          # 一次性：切块 + 嵌入 + 存索引
python stage5/rag.py --q "什么是数据血缘？"
python stage5/rag.py                   # 交互模式
python stage5/rag.py --q "..." --show  # 同时打印检索到的原文
python stage5/rag.py --q "..." --k 6   # 检索更多段
```

## 实测结果

索引：9423 个文本块（来自 5318 篇 md）。

提问"数据资产建设和本体建设是什么关系？"：
- 检索精准命中正主《数据资产建设方法与本体关系》(相似度 0.753)
- 答案忠实复述真实原文："数据资产建设是本体建设的前置工程和治理底座……"
- 出处：真实文件 `domains/data-assets/01-construction-method-and-ontology-relationship.md`

对比阶段④微调：微调会**编造**文件路径和细节；RAG 的答案**基于当场检索的真实原文，可溯源**。

## 管线（三步）

```
建索引(一次性):  wiki → 按小节切块 → bge嵌入成向量 → 存 index.npz + meta.json
问答(每次):      问题 → 嵌入 → 余弦相似度检索 top-k → 拼进提示 → 0.5B 生成 + 出处
```

## 核心概念

### 嵌入 (embedding)
用 `bge-small-zh-v1.5` 把每段文字变成一个 512 维向量，**语义相近的文字，向量也相近**。
这是"按意思检索"而非"按关键词检索"的基础——问"血缘"也能找到讲"lineage"的段落。

### 向量检索（余弦相似度）
向量归一化后，问题向量和每个块向量做点积（=余弦相似度），取最高的 top-k。
本项目用 numpy 手算（9423 个向量瞬间算完），透明可懂；生产环境用 FAISS 等加速，原理相同。

### 切块 (chunking)
按 markdown 小节（`## `）切，语义干净，每块天然带"文档标题 + 小节名"做出处。
过长小节再按长度切。

### RAG vs 微调（什么时候用哪个）
| 需求 | 用什么 |
|---|---|
| 改变模型的风格、语气、格式 | 微调（阶段④）|
| 准确回答、给真实出处、知识随文档更新 | **RAG（本阶段）** |
| 两者兼顾 | 微调调风格 + RAG 保事实 |

## 局限与可改进

- 生成用的是 0.5B 小模型，综合能力有限——换 3B/7B 答案会更好（检索部分不变）。
- 检索可加 **rerank**（先粗检再精排）、**查询改写**、**多路召回**进一步提升。
- 当前每次重建全量索引；可做增量更新。

## 文件
```
build_index.py   切块 + 嵌入 + 存索引（嵌入函数被 rag.py 复用）
rag.py           检索 + 生成 + 出处（交互式 / --q 单次）
index.npz        向量索引（不进 git，可重建）
meta.json        块的元信息+正文（含 wiki 内容，绝不进 git）
```

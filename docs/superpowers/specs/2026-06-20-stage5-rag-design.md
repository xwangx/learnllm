# 阶段⑤ 设计：给数据治理 wiki 挂 RAG

> 学习路径第 5 阶段。真正解决用户最初目标"让模型更懂数据治理"——
> 准确回答、给真实出处、wiki 更新即生效。RAG 是正解（微调不是）。

## 核心管线

```
建索引(一次性):  wiki文档 → 按小节切块 → bge嵌入成向量 → 存 index.npz + meta.json
问答(每次):      问题 → 嵌入 → 余弦相似度检索 top-k 段 → 拼进提示 → 0.5B 生成 + 附出处
```

## 决策（已确认）

- 生成模型：复用本地 Qwen2.5-0.5B（零下载；RAG 中模型只需读懂检索原文，门槛低于凭记忆答）。
- 嵌入模型：bge-small-zh-v1.5（中文向量，约 95MB，ModelScope 下载）。
- 向量检索：numpy 手算余弦相似度（透明可懂；生产才换 FAISS，原理同）。

## 产出物

```
stage5/
├── build_index.py   读 wiki → 按 ## 小节切块 → bge 嵌入 → 存 index.npz + meta.json
├── rag.py           问答主程序：检索 + 生成 + 显示出处（交互式 / --q 单次）
└── README.md        RAG 原理、嵌入、向量检索、RAG vs 微调
```

## 切块策略

- 按 markdown 小节（`## ` 标题）切块——语义干净，天然带"文档标题 + 小节名"做出处。
- 过长小节再按长度切（如每块 ≤ 500 字符，可留少量重叠）。
- 去 frontmatter；跳过自动生成/元数据型内容（复用 stage4 的过滤思路）。
- 每块元信息：源文件路径、文档标题、小节名、正文。

## 嵌入

- 用 transformers 直接加载 bge，取 [CLS] 向量并 L2 归一化（透明，不引入 sentence-transformers）。
- 查询侧按 bge 建议加指令前缀："为这个句子生成表示以用于检索相关文章："。
- 文档侧直接编码。

## 检索 + 生成

- 余弦相似度（向量已归一化 → 点积即可）选 top-k（默认 4）。
- 提示模板：
  > 根据以下参考资料回答问题。若资料未提及，请说"资料中未提到"。
  > 【参考资料】{检索段落，带编号}
  > 【问题】{question}
- 生成后列出每段来源（文件 + 小节 + 相似度分数）。

## 成功标准

- build_index.py 生成索引文件。
- rag.py 提问能检索到真正相关段落、基于其作答、列出出处。
- 对比 stage4：同一问题，RAG 答案基于真实原文、可溯源，而非模型瞎编。
- README 讲清 RAG 原理与 RAG vs 微调。

## 范围之外（YAGNI）

- 不做 Web UI（先 CLI）。
- 不做 rerank、多轮对话、查询改写等高级优化。
- 不做增量索引（重建即可）。

# 吴恩达《Machine Learning Yearning》× 本项目实战对照

把吴恩达的免费小书 **《Machine Learning Yearning》**（中译《机器学习训练秘籍》/《渴望机器学习》）
的方法论，和本项目每个动手阶段结合起来。

## 两者为什么互补

| | 教什么 | 形式 |
|---|---|---|
| **本项目（stage1~6）** | 模型**怎么实现**：反向传播、CNN、注意力、微调、RAG、ViT | 写代码、跑实验 |
| **《ML Yearning》** | 训练这件事**怎么做对**：怎么划数据、怎么诊断、往哪改进 | 策略方法论，不讲数学 |

一句话：**我们造引擎，他教你怎么开车、怎么修车。** 两者合起来才是完整的工程能力。

---

## 一、数据集划分与评估指标

**书里的观点**
- 把数据分成 **训练集 / 开发集(dev，即验证集) / 测试集**，dev 和 test 要尽量贴近真实使用场景、同分布。
- 用 **单一数字评估指标**，便于快速比较两个模型谁更好。
- dev/test 只要"够大到能区分模型差异"即可。

**本项目对应**
- **阶段②**：MNIST 切成 50000 训练 / 10000 验证 / 10000 测试。验证集训练时监控、测试集最后才用一次。→ `stage2/common.py: get_dataloaders()`
- 单一指标：阶段② 用 **accuracy**；阶段⑤ RAG 用 **相似度分数**。

**可深化**：dev/test 的"分布要贴近真实场景"在我们的标准数据集上没体现；真实项目里要特别注意。

---

## 二、先快速搭第一版，再迭代

**书里的观点**
- 不要一开始就追求完美。**先快速搭一个能跑的系统**，再用"误差分析"驱动改进，迭代前进。

**本项目对应**
- 每个阶段都是"先跑通最小可用版，再优化"：
  - 阶段⑤ 先用 numpy 手算检索跑通 RAG，而不是一上来就上 FAISS。
  - 阶段④ 先续写式微调，发现不会答题，再迭代成指令式。**这就是一次典型的"跑通→诊断→改进"循环。**

---

## 三、误差分析（Error Analysis）

**书里的观点**
- **亲手翻看 dev 集里被分错的样本**，是性价比最高的动作。
- 把错误**归类统计**（如"100 个错里 40 个是把猫看成狗"），据此决定优先改哪一类。
- "天花板分析"：算出修好某类错最多能涨多少分，再决定值不值得做。

**本项目对应**
- **阶段②**：`show_mistakes()` 打印被错分的数字（看到 CNN 常把 3↔8、4↔9、9↔7 搞混）。→ 这正是"亲手看错分样本"。

**可深化**：我们只是打印了，没做**归类统计 + 天花板分析**。阶段⑥可以补上（统计 ViT 主要错在哪几类）。

---

## 四、偏差 vs 方差（全书的核心）

**书里的观点**
- 用两个数诊断：**训练误差**、**开发集误差**。
  - 训练误差就高 → **高偏差（欠拟合）**：模型太弱/没学好。
  - 训练误差低但开发误差高 → **高方差（过拟合）**：模型在死记训练集。
- 进一步和**最优误差（贝叶斯误差）/ 人类水平**比，区分"**可避免偏差**"和"方差"。

**本项目对应**
- **阶段② 过拟合演示**（`stage2/overfit_demo.py`）就是活教材：
  - 训练 acc 冲到 100%、验证卡在 89% → 典型 **高方差**。
  - 我们打印的"**过拟合间隙**"= 训练 acc − 验证 acc，正是方差的直接度量。
  - "验证 loss 触底后回升"的剪刀差 = 书里说的过拟合信号。

---

## 五、降低偏差 / 降低方差的手段

**书里的药方**

| 高偏差（欠拟合）怎么治 | 高方差（过拟合）怎么治 |
|---|---|
| 更大/更强的模型 | 加更多训练数据 |
| 训练更久、更好的优化器 | 正则化（L2、dropout） |
| 换更适合的网络结构 | 数据增强 |
| 减少不当的正则化 | 早停（early stopping） |
| | 换更小的模型/特征 |

**本项目对应**
- 降偏差换架构：**MLP → CNN → Transformer/ViT**（阶段②③⑥），一步步用更强的模型。
- 降方差：阶段② 用 **Dropout**；阶段⑥ ViT 用 **数据增强**；过拟合演示里提到 **早停**。
- 阶段④ **LoRA**：用更少参数训练，本身也是一种抑制过拟合的思路。

---

## 六、和人类水平 / 最优误差对标

**书里的观点**
- 人类水平常被当作**贝叶斯误差的估计**。模型离人类水平的差距 = 可避免偏差；训练集到开发集的差距 = 方差。

**本项目对应（可在阶段⑥落地）**
- MNIST：人类几乎 100%，所以阶段② MLP 的 97.5% 仍有"可避免偏差"空间，CNN 的 99.1% 已逼近上限。
- CIFAR-10：人类约 **94%**。阶段⑥ 训出的 ViT/CNN 可以和这个标尺比，判断瓶颈在偏差还是方差。

---

## 七、训练集与开发集分布不一致

**书里的观点**
- 当训练数据和真实场景分布不同时，引入 **training-dev set**（从训练分布里留出、不参与训练的一份）来区分"是方差问题还是数据不匹配问题"。

**本项目对应**
- 我们用的都是同分布标准数据集，没遇到这个问题。但这是真实工程里的高频坑，值得记住。

---

## 八、迁移学习 / 端到端学习

**书里的观点**
- **迁移学习**：在大数据上预训练，再迁移到小数据任务——数据少时极有用。
- 端到端 vs 流水线：要权衡。

**本项目对应**
- **阶段④ 微调 Qwen2.5** = 迁移学习的实战（站在预训练模型肩上）。
- **阶段⑥ ViT** 会亲身验证书里的论断：**ViT 数据饥渴**，没有大规模预训练时在 CIFAR-10 上打不过 CNN——正是"迁移学习/预训练有多重要"的反面教材。

## 把方法论用起来：阶段⑥ 的诊断计划

阶段⑥（从零 ViT vs CNN on CIFAR-10）将刻意按吴恩达的框架做分析：

1. **算偏差/方差**：列出 ViT 与 CNN 各自的 训练误差、验证误差。
2. **和人类水平(~94%)对标**：判断各自瓶颈是"可避免偏差"还是"方差"。
3. **误差分析**：统计 ViT 主要错在哪几类图（归类，不只是打印）。
4. **下结论**：用 bias/variance + 数据量解释"为什么 ViT 需要大数据/预训练"。

这样阶段⑥就是"**实现（写 ViT）+ 方法论（吴恩达式诊断）**"的合体。
